1. Overview

This document outlines two options for testing the Grant Management System before it goes into production use, followed by the deployment plan for Microsoft Azure. The testing phase allows Citrin Cooperman's team to thoroughly evaluate every feature, scrutinize security, and request any modifications before the system is used with real client data.

Key Principle: No real client data will be used during testing. All testing will be performed with sample/dummy data specifically created for evaluation purposes. Real data transfer (if applicable) occurs only after all testing is complete and the system is formally approved.

Option 1

Test on the Current Development Domain

The GMS is currently running on a development domain used during the building phase. Citrin Cooperman's team can access this existing deployment for testing and evaluation.

How It Works

  1. IT team grants access: Citrin Cooperman's IT team allows the testing team to open the development domain link (may require firewall/proxy exception or VPN access depending on network policies)
  2. Test accounts are created: The developer creates test accounts for each role (Superadmin, Manager, Staff, Client) with sample data pre-loaded
  3. Testing begins: The team tests all features, workflows, and security measures using the provided accounts
  4. Feedback is collected: Testing team documents findings, change requests, and security observations
  5. Changes are implemented: Developer implements requested modifications on the development server
  6. Re-testing: Team re-tests after changes are applied
  7. When approved: System is deployed to Microsoft Azure for production use

IT Team Action Required

The IT team needs to whitelist/allow the development domain URL so that the testing team can access it from Citrin Cooperman's network. This may involve:

  • Adding an exception to the web content filter/proxy
  • Allowing the domain through the corporate firewall
  • Or providing VPN access to reach the development server
Option 2

Deploy to Microsoft Azure First, Then Test

The GMS is deployed directly to Citrin Cooperman's Microsoft Azure environment before any testing begins. All testing, scrutiny, and evaluation happens on Citrin Cooperman's own infrastructure.

How It Works

  1. IT team creates an Azure account/subscription: Citrin Cooperman's IT team provisions an Azure subscription (or uses an existing one) for hosting the GMS
  2. IT team provides credentials: IT provides the developer with access credentials (Azure portal login or deployment credentials) to the provisioned Azure resources
  3. Developer deploys the application: The developer deploys the GMS to Azure (web hosting, database, and file storage) and configures all services
  4. Test accounts are created: Developer creates test accounts with sample data on the Azure deployment
  5. Testing begins: The team tests all features directly on Citrin Cooperman's Azure infrastructure
  6. Feedback & changes: Testing team documents findings; developer implements changes directly on the Azure deployment
  7. When approved: The same Azure deployment becomes the production environment (no additional transfer needed)

IT Team Action Required

  • Provision an Azure subscription (if not already available)
  • Create the required Azure resources (or grant the developer permissions to create them):
    • Azure Web Hosting (for the GMS application)
    • Azure Database for MySQL (where all grant data is stored)
    • Azure Cloud File Storage (documents, PDFs, attachments)
    • Azure Key Vault — secure credential storage (recommended)
  • Provide the developer with deployment credentials (for publishing updates to Azure)
  • Configure the custom domain and SSL certificate (can be done later, before go-live)

Option 1 — Advantages

  • Start testing immediately
  • No Azure costs during testing period
  • Minimal IT involvement needed
  • Low-commitment evaluation
  • Easy to walk away if the system is not approved

Option 1 — Considerations

  • IT must allow access to an external domain
  • Testing environment differs from production
  • Requires a separate deployment to Azure after approval
  • Cannot test Azure-specific security features
  • Some testing may need to be repeated after Azure deployment

Option 2 — Advantages

  • Testing on the actual production environment
  • Full security scrutiny including Azure infrastructure
  • No additional transfer after testing — fastest path to go-live
  • Data never leaves Citrin Cooperman's infrastructure
  • IT team can verify Azure security controls in real-time
  • SOC 2 auditors can review the actual deployment

Option 2 — Considerations

  • Requires upfront Azure provisioning effort from IT
  • Azure costs begin during the testing phase
  • Takes longer to begin testing (setup time needed)
  • Higher initial commitment from IT team

6. Recommendation

Recommended: Option 2 (Deploy to Azure First) — Given that security is the firm's primary concern and the system will ultimately run on Azure, testing directly on Citrin Cooperman's Azure infrastructure provides the highest confidence. The team tests exactly what will be in production. Security scrutiny covers both the application and the infrastructure. There is no second transfer step, and SOC 2 evidence can be gathered from the actual production environment.

However, if the team wants a quick, low-commitment preview before investing in Azure setup, Option 1 can serve as a preliminary evaluation — followed by Option 2 for formal testing and production deployment.

7. Testing Plan (Applies to Both Options)

7.1 Functional Testing

Each module should be tested role-by-role to verify all features work as documented.

Phase 1: User Management & Authentication
  • Create users for each role (Superadmin, Manager, Staff, Client)
  • Test login/logout for each role
  • Test password policy enforcement (minimum length, complexity, history)
  • Test account lockout after failed attempts (verify escalating cool-off periods)
  • Test two-factor authentication setup and verification (Manager and Superadmin)
  • Test two-factor authentication backup codes
  • Test force password reset
  • Test user deactivation and verify access is revoked
  • Test force-logout from admin panel
  • Test session timeout (idle and absolute)
Phase 2: Organization & Grant Setup
  • Create test organizations with different settings
  • Assign staff members to organizations
  • Verify staff can only access assigned organizations
  • Create grants of both types (Restricted and Reimbursable)
  • Test grant lifecycle transitions (Pipeline → Active → Closeout → Closed)
  • Set up IP allowlisting and verify enforcement
  • Configure indirect cost rates and fringe benefit rates
Phase 3: Financial Operations
  • Create budgets with three-layer structure (total → lines → periods)
  • Enter expenses and verify budget calculations
  • Test workbook lifecycle (Open → Under Review → Locked → Approved)
  • Create drawdown requests and test approval workflow
  • Record cash receipts and verify reconciliation
  • Test reimbursable billing (hourly and expense models)
  • Generate invoices and track payments
  • Test disallowed expense flagging
  • Test adjustments and journal entries
  • Verify fiscal year rollover with carryforward
Phase 4: Compliance & Audit
  • Set up compliance requirements with due dates
  • Verify email reminders are sent at correct intervals
  • Test compliance submission workflow
  • Create single audit records and SEFA entries
  • Test audit findings workflow (Open → Response Due → Responded → Resolved → Closed)
  • Test matching requirement tracking
  • Test amendment recording with snapshot capture
  • Export SEFA and verify format
Phase 5: Client Portal
  • Test client onboarding wizard
  • Verify permission tiers (Summary Only, Standard, Full Detail)
  • Test each granular permission toggle
  • Test client dashboard data accuracy
  • Test document upload and viewing (as client)
  • Test expense submission (as client)
  • Test messaging between client and staff
  • Verify clients cannot access other organizations' data
Phase 6: Reporting & Export
  • Generate each report type with test data
  • Test filters (date range, organization, grant, category)
  • Export to Excel and verify data accuracy
  • Export to PDF and verify formatting
  • Test portfolio dashboard (cross-client views)
  • Test global search across all entity types
  • Test funders directory
  • Verify export events are logged in the audit trail
Phase 7: Pipeline & Business Development
  • Create pipeline opportunities at each stage
  • Test stage transitions and history tracking
  • Verify pipeline analytics (total value, success rate)
  • Test conversion from pipeline to active grant
  • Test client portal pipeline visibility (if enabled)

8. Security Testing & Scrutiny Plan

This section is designed for the security review team. Each item should be tested and verified. Results should be documented with pass/fail status and any observations.
8.1 Authentication Security
  • Verify brute-force lockout activates after 5 failed attempts (test with wrong passwords)
  • Verify progressive lockout escalation (5 min → 15 min → 30 min → 60 min)
  • Verify lockout applies by both IP and email independently
  • Attempt to log in with a deactivated account (should fail)
  • Verify failed login attempts are logged with IP address and reason
  • Test two-factor authentication enforcement (verify it cannot be bypassed)
  • Verify backup codes work and are single-use
  • Verify the same verification code cannot be reused
8.2 Session Security
  • Verify session identifier changes after login (a new session is issued upon sign-in)
  • Leave session idle for 30+ minutes — verify auto-logout
  • Use a session for 8+ hours — verify forced re-login
  • Attempt to copy a session to a different browser or device — verify the system blocks it
  • Deactivate a user while they are logged in — verify access revoked within 5 minutes
  • Test force-logout from admin panel — verify immediate session termination
  • Verify active sessions are visible in the admin dashboard with IP and device info
8.3 Access Control
  • As Staff, attempt to access a non-assigned organization's data (should be blocked)
  • As Client, attempt to access another organization's data via URL manipulation
  • As Staff, attempt to access superadmin-only pages (should be blocked)
  • As Client, attempt to access admin pages (should be blocked)
  • Revoke a staff member's organization access — verify they lose access within 5 minutes
  • Test IP allowlisting — access from a non-allowed IP (should be blocked and logged)
  • Test client permission tiers — verify each permission toggle works correctly
8.4 Data Security
  • Attempt unauthorized data access through form fields (should be blocked by security safeguards)
  • Attempt script injection in text fields (should be blocked by security safeguards)
  • Attempt unauthorized form submission from an external website (should be blocked by form validation tokens)
  • Verify secure connections (HTTPS) are enforced (no unencrypted access)
  • Verify security headers are present in all responses (use browser inspection tools)
  • Verify uploaded files are validated (try uploading a non-allowed file type)
  • Verify file size limits are enforced
8.5 Audit Trail Verification
  • Perform a data change and verify it appears in the audit log with old/new values
  • Log in from a different device — verify the new IP address is captured
  • Generate a report — verify the export event is logged
  • Access the third-party integration — verify the access is logged
  • Export the audit log to Excel — verify completeness
  • Verify audit records cannot be edited or deleted through the application
  • Verify login history shows IP address, browser/device information, and failure reason for failed attempts
8.6 Third-Party Integration Security
  • Attempt third-party access without credentials (should be denied)
  • Attempt access with expired credentials (should be denied)
  • Attempt to exceed the access rate limit (100 requests per hour — should be throttled)
  • Verify third-party access is read-only (attempts to modify data should be denied)
  • Verify third-party access respects organization boundaries (cannot access other organizations' data)

9.2 Deployment Steps

Step 1 Provision Azure Resources

Responsibility: IT team provisions, developer configures

Azure Service Configuration Est. Monthly Cost
Azure Web Hosting Modern hosting environment, appropriately sized based on usage needs $70–$140
Azure Database for MySQL Entry-level configuration, scalable as usage grows $15–$30
Azure Cloud File Storage Standard tier with local or geographic redundancy $5–$20
Azure Key Vault (secure credential storage) Standard tier ~$0.03
Azure Monitor (optional) Basic tier $0–$10
Estimated Total Monthly Cost $90–$200

Costs are approximate and scale with usage. Production workloads may require higher tiers as user count and data volume grow.

Step 2 Configure Database

  • Create the database on Azure
  • Set character encoding to support all languages and special characters
  • Configure firewall rules to allow only the application's server to connect
  • Enable encrypted connections to the database
  • Load the database structure and initial reference data
  • Create the initial Superadmin account

Step 3 Configure Storage

  • Create a cloud file storage container for document uploads
  • Configure secure access tokens for file retrieval (time-limited, authorized access only)
  • Set access level to "Private" (no anonymous access)
  • Enable file recovery protection (30-day retention recommended)

Step 4 Deploy Application Code

  • Deploy the application to Azure
  • Configure the application environment settings
  • Set application root to the correct directory
  • Configure web server routing rules

Step 5 Configure Secrets (Azure Key Vault)

  • Store database credentials in Azure Key Vault (secure credential storage)
  • Store session security key in Azure Key Vault
  • Store email service credentials in Azure Key Vault
  • Store scheduled task credentials securely in Azure Key Vault
  • Configure the application to securely retrieve credentials from Azure Key Vault
  • No credentials stored in application code or configuration files

Step 6 Set Up the Database

  • Set up the database structure in the correct order
  • Verify all data tables and relationships are created correctly
  • Insert system default settings
  • Create initial Superadmin account with forced password change

Step 7 Configure Custom Domain & SSL

  • Add custom domain to the Azure hosting service (e.g., gms.citrincooperman.com)
  • Configure the domain name to point to the Azure hosting service
  • Enable security certificate (free Azure managed certificate or firm's existing certificate)
  • Enforce secure-only access (automatically redirect all traffic to encrypted connections)
  • Update application configuration with the production URL

Step 8 Security Hardening

  • Network Security: Configure network access rules to restrict traffic to secure connections only
  • Database firewall: Allow only the application server to connect; block all public access
  • Cyberattack Protection: Enable the included tier or upgrade to the enhanced tier
  • Microsoft Defender for Cloud: Enable for security posture assessment
  • Diagnostic logging: Enable application logs and Azure Monitor
  • Backup schedule: Configure Azure Backup with daily database snapshots and 30-day retention
  • Geo-redundancy: Enable geo-redundant storage for Azure Cloud File Storage (data replicated across regions)
  • Access restrictions: Optionally restrict application access to Citrin Cooperman's office network addresses during testing

Step 9 Verification & Go-Live

  • Run the full security testing checklist (Section 8)
  • Verify all features function correctly on Azure
  • Verify email notifications are sending correctly
  • Verify document upload/download works with Azure Cloud File Storage
  • Verify backup/restore functionality
  • Performance test under expected load
  • Final security review sign-off
  • Go-Live — switch DNS to production URL and notify users

11. Post-Deployment Verification

  • Application loads correctly on the custom domain with HTTPS
  • Login/logout works for all roles
  • Two-factor authentication setup and verification works
  • Session timeouts function correctly
  • Document upload and download via Azure Cloud File Storage works
  • Email notifications are being sent and received
  • Database backups are running on schedule
  • Audit logs are recording all actions
  • IP allowlisting enforcement is working
  • Security headers are present in all page responses (verified via browser inspection)
  • Azure Monitor is collecting metrics
  • Microsoft Defender for Cloud shows no critical findings
  • All exports (Excel, PDF) generate correctly
  • Performance is acceptable under normal load
