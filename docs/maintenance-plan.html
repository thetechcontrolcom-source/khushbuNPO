How Azure Takes Care of the Heavy Lifting

By hosting the Grant Management System on Microsoft Azure (Microsoft's cloud platform), the firm no longer needs to worry about maintaining physical servers, installing security updates, or managing network protection. Microsoft handles all of that automatically. This means the internal team can focus on what matters most — managing grant data, supporting users, and keeping the system aligned with the firm's needs.

The table below breaks down exactly who is responsible for each part of the system:

What It Is Who Takes Care of It How It Works
Physical Servers & HardwareMicrosoft AzureFully managed by Microsoft — no action needed from us
Security Updates & Patches (operating system)Microsoft AzureAutomatically applied — keeps the system protected against known vulnerabilities
Database Software Updates (the system that stores all grant data)Microsoft AzureMinor updates applied automatically — ensures the database stays current and secure
Website Security Certificates (SSL/TLS — the padlock in the browser)Microsoft AzureAuto-renewed — ensures all data is encrypted during transmission
Network Security & Protection Against AttacksMicrosoft AzureBuilt into Azure's global infrastructure — always on
Application Features & Business Logic (the GMS system itself)System Owner (Khushbu Shah)Updated and deployed through a controlled, version-tracked process
Data Accuracy & Backups (grant records, financial data, documents)Shared — Azure + System OwnerAzure runs automated backups; system owner verifies data integrity
User Accounts & Access (who can log in and what they can see)Internal Staff / Super AdminManaged through the GMS admin panel
In simple terms: Microsoft handles the "building and plumbing" (servers, security, backups). We handle the "furniture and layout" (the application, data, and user access). This is a standard responsibility model used across the industry.

Azure Services We Use

The Grant Management System uses several Azure services, each serving a specific purpose. Here's what each one does in plain terms:

Azure Service What It Does How It's Maintained
Azure App Service
(where the GMS application runs)		 Hosts the Grant Management System so staff and clients can access it through their web browser Microsoft manages the underlying servers; the application can handle traffic increases automatically
Azure Database for MySQL
(where all grant data is stored)		 Stores all grant records, financial data, user accounts, budgets, drawdowns, compliance records, and audit trails Azure handles daily backups automatically; data can be restored to any point in time if something goes wrong
Azure Blob Storage
(where files and documents live)		 Stores uploaded grant documents, PDFs, funder correspondence, award letters, budget attachments, and expense receipts Files are stored with built-in redundancy so they're never lost; older files can be archived to save costs
Azure Monitor + Application Insights
(health monitoring & alerts)		 Watches the system around the clock — tracks response times, detects errors, and sends alerts if anything looks unusual Runs continuously in the background; no manual effort needed — alerts come via email when attention is required
Azure Key Vault
(secure password & credential storage)		 Safely stores sensitive information like database passwords and service credentials — similar to a secure digital safe Credentials can be rotated (changed) on a schedule without disrupting the system
Microsoft Entra ID (optional)
(single sign-on for staff)		 Allows staff to log into GMS using their existing firm credentials — no separate password to remember Managed through the firm's existing identity system; no additional maintenance required
If the firm qualifies for Microsoft Tech for Social Impact, Microsoft provides up to $3,500 per year in free Azure credits for eligible nonprofit organizations. This could significantly reduce or fully offset hosting expenses.

Routine Maintenance Schedule

One of the key benefits of using Azure is that many maintenance tasks happen automatically. Below is a breakdown of what's handled on each cycle and what requires manual attention.

Daily — Automated by Azure
  • Full database backup (all grant data, financial records, user accounts)
  • Security certificate monitoring (ensures encrypted connections stay active)
  • System health checks (verifies the application is running and responsive)
  • Error tracking (any issues are automatically logged for review)
Weekly — Quick Review
  • Review the monitoring dashboard (check for any trends or warnings)
  • Check error rates (make sure nothing unusual has occurred during the week)
  • Check file storage usage (grant documents, PDFs, attachments)
  • Confirm daily backups completed successfully
Monthly — Detailed Review
  • Review active user accounts (disable unused accounts, verify access levels)
  • Review GMS audit log (who accessed what, when, and any changes made)
  • Test a backup restore (verify that backups can actually be used to recover data)
  • Review compliance calendar (upcoming funder deadlines, report due dates)
  • Review security advisories (check for any known issues that need attention)
  • Review storage costs (make sure usage is within expected range)
Quarterly — Strategic Review
  • Rotate credentials (change database passwords and service keys as a security best practice)
  • Review Azure spending (compare actual costs against budget expectations)
  • User access audit (ensure only the right people have access to the right data)
  • Software version check (ensure the system is running on supported versions)
  • Update documentation (keep the SOP and Interface Guide current)
Annually — Comprehensive Assessment
  • Disaster recovery drill (simulate a worst-case scenario to ensure we can recover quickly)
  • Hosting plan review (evaluate if the current service tier still fits our needs and budget)
  • Security assessment (independent review to identify any potential vulnerabilities)
  • Third-party component review (ensure all external libraries used by the system are up to date)
  • Azure service-level agreement review (confirm Microsoft is meeting their uptime commitments)

Backup & Disaster Recovery

Protecting grant data is a top priority. Azure provides multiple layers of backup and recovery options so that data is never permanently lost — even in the unlikely event of a major system failure.

Protection Feature How It's Set Up What This Means
Automated Daily BackupsFull backup every dayAzure automatically saves a complete copy of all grant data, financial records, and user information — no manual effort needed
Backup Retention PeriodConfigurable: 7 to 35 daysWe can go back and restore data from any day within this window — set through the Azure management portal
Point-in-Time RestoreAny moment within the retention windowIf a record is accidentally deleted or modified, we can restore the database to the exact second before it happened
Geographic Redundancy (recommended)Backup copies stored in a separate regionEven if an entire Azure data center goes offline, our data is safe in a different geographic location
File Storage Redundancy (grant documents, PDFs)Multiple copies maintained automaticallyAll uploaded documents are stored with built-in copies — recommended to use geo-redundant storage for production
Recovery Time (how fast we can be back online)Less than 1 hourIn a worst-case scenario, the system can be restored and operational within one hour
Recovery Point (how much data could be lost)Less than 24 hours of dataAt most, we would lose the last day's entries — which can be re-entered from source documents if needed
A written Disaster Recovery Runbook (step-by-step instructions for restoring the system in an emergency) should be maintained and kept up to date by the system owner. This document ensures that recovery procedures are clear and can be followed by anyone on the team, even without prior experience.

Continuity & Knowledge Transfer

To ensure the system can be maintained even if team members change, the following documentation and resources are maintained:

  • Standard Operating Procedures (SOP) — step-by-step guide for all routine tasks
  • Interface Guide — screenshots and instructions for navigating every screen in the system
  • Version-Controlled Codebase — all code changes are tracked with a full history of who changed what and when
  • Azure Deployment Guide — instructions for publishing updates to the live system
  • This Maintenance Plan — reviewed and updated annually
These documents are designed so that a qualified professional can step in and manage or support the system with minimal ramp-up time. The goal is to avoid any single point of dependency.

Support & Incident Response

When something goes wrong — whether it's a system outage or a minor display issue — it's important to have a clear process for reporting and resolving problems. The table below outlines expected response and resolution times based on severity:

Priority Level What It Means Initial Response Expected Resolution
P1 — Critical The entire system is down and users cannot access it, or there is a risk of data loss Within 1 hour Same business day
P2 — High A key feature is not working (e.g., users can't enter expenses, generate reports, or submit drawdowns) Within 4 hours Within 2 business days
P3 — Medium A secondary feature has an issue, but a workaround is available (e.g., a filter isn't working but data can still be exported manually) Within 1 business day Within 1 week
P4 — Low Minor cosmetic issue or enhancement request (e.g., a label is misaligned, a column could be wider) Within 3 business days Included in next scheduled update
All support requests should be directed to the system owner through internal channels. For Critical (P1) issues that appear to be related to Azure infrastructure (e.g., the entire site is unreachable), the IT Department should also be notified so they can check the Azure service status.
